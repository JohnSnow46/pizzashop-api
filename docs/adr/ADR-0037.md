# ADR-0037: Frontend — iteracja auth (logowanie/rejestracja klienta), token w `localStorage`, `AuthContext`, brak zmian backendowych

**Data:** 2026-07-23
**Status:** Zaakceptowana

**Kontekst.**
Po MVP katalog+koszyk (ADR-0035) i checkoucie gościa (ADR-0036) kolejna iteracja to logowanie/
rejestracja klienta. Backend już to udostępnia: `AuthController` (`/api/auth/register`,
`/api/auth/login`, oba `AllowAnonymous`) zwraca `AuthResultDto(string Token, Guid UserAccountId,
UserRole Role, Guid? CustomerId)` (ADR-0026). `RegisterCustomerCommand(Email, Password, FullName,
PhoneNumber?)` i `LoginCommand(Email, Password)` — walidacja backendu: `Email` format, `Password`
min. 8/max 100 znaków przy rejestracji (login bez dolnego progu), `FullName` wymagane, telefon
opcjonalny. `RegisterCustomerCommand` tworzy `Customer`+`LoyaltyAccount` atomowo i zwraca token
od razu (auto-login). Poza zakresem tej iteracji: punkty lojalnościowe (`PointsToRedeem` nadal
zawsze `null`), historia zamówień/`/me`-profil w UI, live-tracking, panel admina/obsługi — routing
zostawia tylko miejsce. CORS (`Program.cs`, ADR-0035): polityka `"frontend"`, `WithOrigins` z
konfiguracji, `AllowAnyHeader().AllowAnyMethod()`, **bez `AllowCredentials`**.

**Decyzja.**

*(1) Kontrakt: bez zmian backendowych — reużycie istniejącego `AuthController`.* `POST /api/auth/register`
przyjmuje `{ email, password, fullName, phoneNumber: string | null }` → `AuthResultDto`; `POST /api/auth/login`
przyjmuje `{ email, password }` → `AuthResultDto`. `AuthResultDto` niesie JWT (`token`), `userAccountId`,
`role` (string enum: `"Customer"` w tym flow), `customerId` (zawsze wypełniony dla Customer). Frontend
nie tworzy nowych endpointów ani DTO po stronie Api.

*(2) Token w `localStorage`, NIE httpOnly cookie.* Uzasadnienie: projekt już stoi na bezstanowym JWT
przekazywanym nagłówkiem `Authorization: Bearer` (ADR-0026) — backend nigdzie nie ustawia cookies.
Przejście na httpOnly cookie wymagałoby zmiany backendu (ustawianie `Set-Cookie` z `SameSite=None;
Secure` dla cross-origin dev, ochrony CSRF, endpointu wylogowania czyszczącego cookie) — nieproporcjonalne
do zakresu tej iteracji i niespójne z „jedyna zmiana backendowa" z poprzednich ADR. `localStorage` +
ręczny nagłówek `Authorization` **nie wymaga zmiany CORS**: `AllowAnyHeader()` już dopuszcza nagłówek
`Authorization` w żądaniu cross-origin — `AllowCredentials` dotyczy cookies/TLS client certs/HTTP auth,
nie custom-headera ustawianego przez JS. Polityka `"frontend"` w `Program.cs` zostaje bez zmian; tylko
komentarz przy niej (linie ~140–144) jest nieaktualny („no cookies/Authorization header sent cross-origin")
— do skorygowania jako czysto dokumentacyjna poprawka. Świadomy trade-off: token w `localStorage` jest
czytelny dla JS, więc podatny na eksfiltrację przy XSS (inaczej niż httpOnly cookie) — akceptowalne teraz
(brak treści generowanej przez użytkownika renderowanej jako HTML w tym MVP); do rewizji, gdyby doszły
funkcje zwiększające powierzchnię XSS.

*(3) `AuthContext` — wzorzec analogiczny do `CartContext`.* Stan: `{ user: { userAccountId, email, role,
customerId } | null, token: string | null, isAuthenticated: boolean, isLoading: boolean }`. Akcje:
`login(email, password)`, `register(email, password, fullName, phoneNumber)`, `logout()` — wszystkie
`async`, aktualizują stan i `localStorage` (klucz `pizzashop.auth`, jeden JSON: `{ token, user }`) po
sukcesie; `login`/`register` propagują `ApiError` do wywołującego (obsługa błędu w formularzu, jak w
checkoucie ADR-0036 pkt 7). `isLoading` tylko na starcie (odczyt z `localStorage` przy montowaniu, bez
wywołania sieciowego — brak endpointu do walidacji tokenu poza `/me`, którego ta iteracja nie woła
proaktywnie). `AuthProvider` w drzewie **nad** `CartProvider`/`Routes` (np. w `main.tsx` lub `App.tsx`,
poziom wyżej niż `Layout`), żeby `useAuth()` był dostępny z `Layout` i stron.

*(4) Nagłówek `Authorization` dodawany globalnie w `api/client.ts` przez moduł-poziomu setter, nie
per-request.* `client.ts` eksportuje `setAuthToken(token: string | null)` (zmienna modułowa) wywoływane
przez `AuthContext` przy starcie/logowaniu/wylogowaniu; `get`/`post` dołączają
`Authorization: Bearer ${token}` do nagłówków, gdy token ustawiony — bez zmiany sygnatur wywołań w
`menuApi`/`ordersApi`/itd. Unika przekazywania tokenu przez każdy call site i cyklicznego importu
(`client.ts` nie importuje `AuthContext`).

*(5) Wpływ na guest checkout (ADR-0036): minimalny w tej iteracji — bez usuwania trybu gościa.*
`CreateOrderCommand` nadal nie przyjmuje `customerId` w ciele (czytany z `ICurrentUser` po stronie
handlera, ADR-0026) — frontend nic nie zmienia w mappingu. Jedyna zmiana UX: `ContactStep` w
`CheckoutPage` odczytuje `useAuth()` i, gdy `isAuthenticated`, **wstępnie wypełnia** pola kontaktu
(`fullName`, `email` z `user`, telefon pozostaje pusty do ręcznego uzupełnienia — `AuthResultDto` go nie
niesie) — pola pozostają edytowalne, walidacja bez zmian. `PointsToRedeem` pozostaje zawsze `null`.
Historia zamówień/profil klienta to przyszła iteracja.

*(6) Routing: `/login` i `/register` jako nowe publiczne trasy; brak `ProtectedRoute` teraz.* Brak jeszcze
żadnej strony wymagającej zalogowania (brak `/account`, `/orders`) — `ProtectedRoute` odłożony jako
przyszły krok (TODO w kodzie), gdy pojawi się pierwsza taka strona. Dodajemy za to
`RedirectIfAuthenticated` — mały wrapper zawijający `/login` i `/register`: jeśli `isAuthenticated`,
przekierowuje na `/`. Z listy TODO w `routes.tsx` usuwamy `/login`; `/register` dopisujemy obok;
`/orders/:trackingToken` zostaje jako przyszłe.

*(7) `Layout.tsx`: link „Zaloguj" gdy niezalogowany; e-mail klienta + „Wyloguj" gdy zalogowany.* Obok
istniejącego linku koszyka: `!isAuthenticated` → `<Link to="/login">Zaloguj</Link>`; `isAuthenticated` →
tekst z `user.email` (`AuthResultDto` nie niesie `fullName`, więc na start pokazujemy e-mail) + przycisk
„Wyloguj" wołający `logout()` i (opcjonalnie) `navigate('/')`.

**Alternatywy rozważone.**
- *httpOnly cookie* — odrzucona teraz: wymaga zmian backendu (ustawianie cookie, CSRF, `SameSite`/`Secure`
  dla cross-origin dev) nieproporcjonalnych do zakresu; do rozważenia, gdyby XSS stał się realnym ryzykiem.
- *`ProtectedRoute` już teraz „na zapas"* — odrzucone: brak jeszcze stron do ochrony, YAGNI.
- *Endpoint `/me` wołany proaktywnie przy starcie do walidacji tokenu* — odrzucone teraz: dokłada round-trip
  bez realnej korzyści; token traktowany optymistycznie do pierwszego 401 z Api.

**Konsekwencje.**
- Zero zmian w kontraktach/kodzie Api poza jedną poprawką komentarza w `Program.cs` (CORS, pkt 2).
- Rośnie ręczny mirror typów o `auth/types.ts` (`AuthResult`, `LoginRequest`, `RegisterRequest`).
- `api/client.ts` przestaje być bezstanowy (moduł trzyma token w pamięci przez `setAuthToken`).
- `localStorage` jako nośnik tokenu to świadomy, zapisany trade-off bezpieczeństwa (XSS) — do rewizji przy
  wzroście powierzchni ataku.
- Historia zamówień, profil klienta, punkty lojalnościowe, `ProtectedRoute` — nadal poza zakresem.

**Zakres/kroki dla buildera.**

*Poza zakresem buildera teraz:* punkty lojalnościowe (`PointsToRedeem` zawsze `null`), historia zamówień,
profil klienta, `ProtectedRoute`/strony chronione, live-tracking SignalR, panel admina/obsługi, testy/lint
frontendu w CI, generacja typów z OpenAPI.

1. `frontend/src/auth/types.ts` — `AuthResult { token; userAccountId; role: 'Customer' | 'Employee' |
   'RestaurantAdmin' | 'SuperAdmin'; customerId: string | null }`, `LoginRequest { email; password }`,
   `RegisterRequest { email; password; fullName; phoneNumber: string | null }`.
2. `frontend/src/auth/authApi.ts` — `login(req)`/`register(req)` → `apiClient.post('/auth/login'|'/auth/register', req)`.
3. `frontend/src/auth/authStorage.ts` — `saveAuth`/`loadAuth`/`clearAuth` na `localStorage`, klucz
   `pizzashop.auth`, analogicznie do `cartStorage.ts`.
4. `frontend/src/api/client.ts` — moduł-poziomu `setAuthToken(token: string | null)`; `get`/`post`
   dołączają `Authorization: Bearer ${token}`, gdy ustawiony. Zero zmian w sygnaturach.
5. `frontend/src/auth/AuthContext.tsx` — Context (wzorzec `CartContext.tsx`): stan z Decyzji (3), akcje
   `login`, `register`, `logout`; przy montowaniu `loadAuth()` → `setAuthToken`; błędy propagowane jako
   `ApiError` (łapie formularz); `logout()` czyści stan + `clearAuth()` + `setAuthToken(null)`.
6. `frontend/src/hooks/useAuth.ts` — `useContext(AuthContext)` z guardem, jak `useCart.ts`.
7. Montowanie `AuthProvider` — nad `CartProvider`/`AppRoutes` w `main.tsx`/`App.tsx`.
8. `frontend/src/components/auth/RedirectIfAuthenticated.tsx` — `useAuth()`, jeśli `isAuthenticated` →
   `<Navigate to="/" replace />`, inaczej `children`.
9. `frontend/src/pages/LoginPage.tsx` — e-mail/hasło, walidacja ręczna (e-mail format, hasło niepuste),
   submit → `login()`, sukces → `navigate('/')`; 401 → „nieprawidłowy e-mail lub hasło".
10. `frontend/src/pages/RegisterPage.tsx` — e-mail/hasło/imię i nazwisko/telefon (opcjonalny), walidacja
    mirrorująca `RegisterCustomerCommandValidator` (hasło 8–100 znaków, `fullName` wymagane max 200,
    telefon max 30), submit → `register()` (auto-login), sukces → `navigate('/')`; 409 → „konto już istnieje".
11. `frontend/src/routes.tsx` — dodać `/login` (w `RedirectIfAuthenticated`) i `/register` (analogicznie);
    usunąć `/login` z TODO-komentarza.
12. `frontend/src/components/Layout.tsx` — `useAuth()`; link „Zaloguj" / e-mail+„Wyloguj".
13. `frontend/src/checkout/` (`ContactStep`) — przy `isAuthenticated`, wstępnie wypełnić `fullName`/`email`
    z `useAuth().user` (pola nadal edytowalne); brak zmian w `mapCartToOrder.ts`/`CreateOrderCommand`.
14. `src/PizzaShop.Api/Program.cs` — drobna korekta komentarza przy sekcji CORS (linie ~140–144): usunąć
    nieaktualne „no cookies/Authorization header sent cross-origin", dopisać odwołanie do ADR-0037.
    Czysto dokumentacyjne — bez zmiany zachowania/kodu.
