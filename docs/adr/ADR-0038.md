# ADR-0038: Frontend — live-tracking statusu zamówienia (SignalR), nowa trasa `/orders/track/:trackingToken`

**Data:** 2026-07-24
**Status:** Zaakceptowana

**Kontekst.**
Backend ma od Iteracji 4 (ADR-0028, ADR-0031, ADR-0032) w pełni działający live-tracking:
`OrderTrackingHub` (`/hubs/order-tracking`, `[AllowAnonymous]`, grupy per `OrderId`),
`SubscribeToGuestOrder(string trackingToken)` (GUID jako string, cichy brak subskrypcji przy
złym tokenie) i `SubscribeToOrder(Guid orderId)` (JWT wymagany przez `access_token` w query
stringu połączenia — `Program.cs` sprawdza prefiks ścieżki `/hubs/order-tracking`), push
`"OrderStatusChanged"` → `{ orderId, status, estimatedReadyAt }`. REST do stanu początkowego:
`GET /api/orders/{id}` (`[Authorize]`, właściciel/obsługa) i `GET /api/orders/track/{token}`
(`[AllowAnonymous]`, gość). Frontend (`OrderConfirmationPage.tsx`) obecnie tylko wypisuje
`guestTrackingToken` jako tekst, z komentarzem odkładającym live-tracking. Brak `@microsoft/
signalr` w zależnościach; brak jakiejkolwiek infrastruktury testowej frontendu (zero
`vitest`/`jest`/RTL, zero plików `*.test.*`) — stan niezmieniony od ADR-0035/36/37, które też
nie wprowadziły testów mimo nietrywialnej logiki (mapowanie koszyka, wizard checkoutu,
`AuthContext`). Vite proxy (`vite.config.ts`) obecnie przekazuje w dev tylko `/api/*` do Api;
`/hubs/*` nie jest jeszcze proxowane.

**Decyzja.**

*(1) Zależność: `@microsoft/signalr` (klient oficjalny, ten sam ekosystem co backendowy
`Microsoft.AspNetCore.SignalR`).* Dodanie do `dependencies` w `frontend/package.json`.

*(2) Transport: relatywny URL `/hubs/order-tracking`, spójnie z `BASE_URL = '/api'` w
`client.ts`.* W produkcji frontend musi być serwowany z tego samego originu co Api (już
założenie `client.ts`), więc URL huba jako same-origin względna ścieżka działa bez zmian. W
dev — `vite.config.ts` dostaje drugi wpis proxy dla `/hubs` z `ws: true` (WebSocket wymaga
jawnego flagowania w proxy Vite, w przeciwieństwie do zwykłego REST-owego `/api`).

*(3) Jeden hook = jedno połączenie, tworzone i zamykane z cyklem życia komponentu — bez
globalnego singletona połączenia.* Alternatywa (moduł-poziomu singleton `HubConnection`
reużywany między stronami, analogicznie do `authToken` w `client.ts`) została odrzucona:
w tej iteracji tylko jeden ekran naraz subskrybuje jedno zamówienie, więc singleton dokłada
złożoność (zarządzanie wieloma subskrypcjami grup na jednym połączeniu, invalidacja przy
zmianie zamówienia) bez realnej korzyści. Do rewizji, gdyby powstał panel z wieloma
jednocześnie śledzonymi zamówieniami (np. panel obsługi/kuchni) — wtedy singleton + Map
`orderId → handlers` zyskuje sens.

*(4) `HubConnectionBuilder().withUrl(HUB_URL, token ? { accessTokenFactory: () => token } : {})
.withAutomaticReconnect().build()`.* `accessTokenFactory` dołączany **tylko** gdy
`useAuth().token` istnieje — dla ścieżki gościa (`trackingToken`) hub jest `[AllowAnonymous]`
i nie potrzebuje JWT, więc pomijamy `accessTokenFactory` zamiast wysyłać pusty token.
`withAutomaticReconnect()` z domyślnymi opóźnieniami (0/2s/10s/30s) — status realizacji
zamówienia to dane, przy których krótkie zerwanie połączenia (Wi-Fi, roaming) nie powinno
wymagać ręcznego odświeżenia strony przez klienta.

*(5) Hook `useOrderTracking(source)` w `frontend/src/hooks/useOrderTracking.ts`.*

```ts
type OrderTrackingSource = { orderId: string } | { trackingToken: string }

interface UseOrderTrackingResult {
  order: Order | null            // pełny OrderDto (REST), pola status/estimatedReadyAt
                                  // aktualizowane live z pushu; reszta pozostaje z fetchu
                                  // (nie zmienia się w czasie życia trackingu)
  isLoading: boolean              // trwa początkowy fetch REST
  error: string | null            // błąd REST (404/sieć); błędy Huba są celowo ciche (ADR-0028)
  connectionState: 'connecting' | 'connected' | 'reconnecting' | 'disconnected'
}

function useOrderTracking(source: OrderTrackingSource): UseOrderTrackingResult
```

Wybór wariantu (`orderId` vs. `trackingToken`) determinuje zarówno endpoint REST
(`getOrderById`/`getOrderByTrackingToken`), jak i metodę Huba (`SubscribeToOrder`/
`SubscribeToGuestOrder`) — dokładnie te same dwie ścieżki co backend, bez trzeciej,
"zgadywanej" kombinacji. Efekt: 1) fetch REST → `setOrder`, `setIsLoading(false)`,
złapany 404/sieciowy błąd → `setError`; 2) budowa połączenia, `connection.on
('OrderStatusChanged', payload => { if (payload.orderId === order?.id) setOrder(prev => prev &&
({ ...prev, status: payload.status, estimatedReadyAt: payload.estimatedReadyAt })) })`
(porównanie `orderId` to tania asercja spójności grupy, nie mechanizm bezpieczeństwa — grupa
Huba już filtruje po stronie serwera); 3) `connection.start()` → `invoke('SubscribeToGuestOrder',
trackingToken)` albo `invoke('SubscribeToOrder', orderId)`; 4) `onreconnecting`/`onreconnected`/
`onclose` → `connectionState`; 5) cleanup w `useEffect` → `connection.stop()`. Brak
rozróżnienia "token zły" vs. "zamówienie nie istnieje" na poziomie Huba (ADR-0028) — hook nie
próbuje tego odróżniać; jeśli REST fetch się powiódł (order istnieje/token poprawny), a mimo
to nigdy nie przyjdzie push, UI po prostu pozostaje na ostatnim znanym stanie z REST-a
(zazwyczaj `PendingAcceptance`) — akceptowalne, spójne z resztą kontraktu.

*(6) Komponent prezentacyjny `OrderTrackingStatus` w
`frontend/src/components/orders/OrderTrackingStatus.tsx`.* Przyjmuje `{ order, isLoading, error,
connectionState }` (dokładnie `UseOrderTrackingResult`) — czysto prezentacyjny, nie woła hooka
sam, żeby dało się go użyć i przetestować niezależnie od transportu. Renderuje: polski label
statusu (mapa `OrderStatus → string`, np. `PendingAcceptance` → "Oczekuje na potwierdzenie",
`OutForDelivery` → "W drodze", `Rejected` → "Odrzucone" itd.), `estimatedReadyAt` sformatowany
lokalnie (jeśli obecny), oraz mały wskaźnik połączenia ("Połączono na żywo" / "Łączenie…" /
"Ponowne łączenie…" / "Rozłączono — dane mogą być nieaktualne").

*(7) Nowa trasa `/orders/track/:trackingToken` — TAK, dodajemy.* Uzasadnienie: `CLAUDE.md`
wymaga wprost "bezpiecznego, nieodgadnalnego identyfikatora do śledzenia statusu (np. GUID w
URL)" dla gościa — dziś token istnieje tylko jako tekst na stronie potwierdzenia, czytany z
`sessionStorage`. To niewystarczające z dwóch powodów niezależnych od tego, że
`OrderConfirmationPage` technicznie *ma już* `orderId`+token po checkoucie: (a)
`sessionStorage` nie jest współdzielony między kartami/urządzeniami — gość, który zamknie
kartę, otworzy link na telefonie, albo prześle status komuś innemu, nie ma żadnego
adresowalnego URL-a do tego; (b) sam tekst tokenu bez URL-a wymagałby, żeby gość ręcznie
wklejał GUID w jakiś formularz — gorszy UX niż jeden klikalny/kopiowalny link. Trasa mapuje
się 1:1 na istniejący endpoint REST `GET /api/orders/track/{trackingToken}` i wariant
`{ trackingToken }` hooka — zero nowego kodu backendowego. `OrderConfirmationPage` dodatkowo
pokazuje ten link (nie tylko surowy token) jako klikalny/kopiowalny adres, żeby gość mógł go
od razu zapisać/przesłać dalej.

*(8) Testy: NIE wprowadzamy Vitest/RTL w tym zadaniu.* Trzy poprzednie iteracje frontendowe
(ADR-0035/36/37) obsłużyły porównywalnie nietrywialną logikę (mapowanie koszyka na
`CreateOrderCommand`, wieloetapowy wizard checkoutu, `AuthContext` z persystencją) bez
wprowadzania jakiegokolwiek test runnera — to spójny, świadomy precedens w tym repo, nie
przeoczenie. Wprowadzenie Vitest wyłącznie pod jeden hook w tym zadaniu byłoby: (a)
niespójne z zakresem — nikt nie zlecił decyzji o strategii testów frontendu jako takiej; (b)
wymagałoby ustaleń wykraczających poza ten ADR (integracja z CI, `npm test` w
`.github/workflows/ci.yml`, konwencja nazewnictwa testów frontendowych) — to osobna decyzja,
nie skutek uboczny jednej funkcjonalności. Mitigacja: hook projektowany jako cienki (logika
mapowania wariantu → wywołanie REST/Huba, merge pushu w stan) i explicite oddzielony od
komponentu prezentacyjnego (Decyzja 6) właśnie po to, żeby dało się go pokryć testami later
bez przepisywania, gdy/jeśli zapadnie osobna decyzja o wprowadzeniu Vitest dla frontendu.

**Alternatywy rozważone.**
- *Singleton `HubConnection` reużywany między stronami* — odrzucone teraz (Decyzja 3), do
  rewizji przy panelu z wieloma śledzonymi zamówieniami jednocześnie.
- *Brak nowej trasy, tracking tylko z `sessionStorage` na stronie potwierdzenia* — odrzucone:
  łamie wymóg "GUID w URL" z `CLAUDE.md` i nie przetrwa nowej karty/urządzenia (Decyzja 7).
  Serwer widziałby wtedy identyczny ruch (REST+Hub), różnica jest wyłącznie w tym, skąd
  frontend bierze token — więc koszt dodania trasy jest czysto frontendowy.
- *Polling REST zamiast SignalR* — odrzucone: backend już ma gotowy, działający kanał
  push (ADR-0028/31/32); polling dokładałby zbędne opóźnienie i ruch bez korzyści.
- *Wprowadzenie Vitest już teraz* — odrzucone (Decyzja 8), do rewizji jako osobna decyzja o
  strategii testów frontendu, nie w ramach tego ADR.

**Konsekwencje.**
- Nowa zależność runtime: `@microsoft/signalr`.
- `vite.config.ts` zyskuje drugi wpis proxy (`/hubs`, `ws: true`) obok `/api`.
- Nowa trasa publiczna `/orders/track/:trackingToken`; TODO w `routes.tsx` o przyszłym
  trackingu znika (zrealizowane).
- `api/types.ts` rośnie o mirror `OrderDto`/`OrderItemDto`/`OrderItemExtraDto`/`OrderStatus`/
  `PaymentStatus` oraz payload `OrderStatusChanged` — przyda się też przyszłej historii
  zamówień (poza zakresem teraz, ADR-0037 pkt "poza zakresem").
- Brak zmian backendowych — czysta konsumpcja istniejącego kontraktu (ADR-0028/31/32).
- Frontend nadal bez infrastruktury testowej — świadomy, zapisany trade-off (Decyzja 8), nie
  przeoczenie.

**Zakres/kroki dla buildera.**

*Poza zakresem teraz:* panel obsługi/kuchni z wieloma śledzonymi zamówieniami jednocześnie,
historia zamówień klienta, Vitest/RTL i jakakolwiek zmiana w CI, i18n (teksty na sztywno po
polsku, jak reszta UI).

1. `frontend/package.json` — dodać `@microsoft/signalr` do `dependencies`.
2. `frontend/vite.config.ts` — dodać wpis proxy `'/hubs': { target: 'http://localhost:5105',
   changeOrigin: true, ws: true }` obok istniejącego `/api`.
3. `frontend/src/api/types.ts` — dopisać mirrory: `OrderStatus` (union 8 wartości, patrz
   `src/PizzaShop.Domain/Enums/OrderStatus.cs`), `PaymentStatus` (union 5 wartości, patrz
   `src/PizzaShop.Domain/Enums/PaymentStatus.cs`), `OrderItemExtra` (patrz
   `OrderItemExtraDto.cs`), `OrderItem` (mirror `OrderItemDto`), `Order` (mirror `OrderDto`,
   `src/PizzaShop.Application/Orders/Dtos/OrderDto.cs` — reużyć istniejące `ContactDetails`/
   `Address`/`FulfillmentType`/`PaymentMethod`/`Money`), `OrderStatusChangedEvent { orderId:
   string; status: OrderStatus; estimatedReadyAt: string | null }` (mirror payloadu
   `SignalROrderNotifier`). Komentarz nad każdym wg konwencji pliku ("Mirror of ...").
4. `frontend/src/api/ordersApi.ts` — dodać `getOrderById(orderId: string): Promise<Order>`
   (`GET /orders/{id}`, wymaga JWT — dołączany automatycznie przez `client.ts`) i
   `getOrderByTrackingToken(trackingToken: string): Promise<Order>` (`GET /orders/track/
   {trackingToken}`, `AllowAnonymous`).
5. `frontend/src/hooks/useOrderTracking.ts` — hook z Decyzji 5. Sygnatura, zachowanie i
   kształt zwracany dokładnie jak w Decyzji 5. Stała `HUB_URL = '/hubs/order-tracking'` u góry
   pliku, komentarz analogiczny do `BASE_URL` w `client.ts` (odniesienie do Decyzji 2 tego
   ADR). Token JWT do `accessTokenFactory` przez `useAuth().token` — bez dotykania
   `client.ts`/`setAuthToken` (ten kanał jest tylko dla `fetch`, nie dla `HubConnection`).
6. `frontend/src/components/orders/OrderTrackingStatus.tsx` — komponent z Decyzji 6, czysto
   prezentacyjny (props = `UseOrderTrackingResult`), mapa etykiet polskich dla wszystkich 8
   wartości `OrderStatus` (w tym `Rejected`/`Cancelled` jako stany końcowe negatywne —
   odróżnić wizualnie/tekstowo od `Completed`).
7. `frontend/src/pages/OrderConfirmationPage.tsx` — zastąpić akapit z surowym tokenem: dla
   gościa (`result.guestTrackingToken` niepuste) wywołać `useOrderTracking({ trackingToken:
   result.guestTrackingToken })`, wyrenderować `OrderTrackingStatus` oraz klikalny/kopiowalny
   link do `/orders/track/{token}` (Decyzja 7); dla zalogowanego (`guestTrackingToken === null`)
   wywołać `useOrderTracking({ orderId: result.orderId })` i ten sam komponent. Zaktualizować
   komentarz nagłówkowy pliku (referencja do ADR-0038 obok ADR-0036).
8. `frontend/src/pages/TrackOrderPage.tsx` — nowa strona: `useParams<{ trackingToken: string
   }>()`, `useOrderTracking({ trackingToken })`, renderuje numer zamówienia (z `order.number`
   po udanym fetchu) + `OrderTrackingStatus`; stan pustki gdy `error` (np. zły/niezależny
   token — REST zwróci 404) — komunikat "Nie znaleziono zamówienia o tym identyfikatorze" bez
   ujawniania, czy token jest źle sformatowany czy po prostu nieznany (spójne z ciszą Huba,
   ADR-0028).
9. `frontend/src/routes.tsx` — dodać `<Route path="/orders/track/:trackingToken"
   element={<TrackOrderPage />} />`; usunąć komentarz `TODO: /orders/:trackingToken —
   przyszła iteracja`; zaktualizować komentarz nagłówkowy pliku (dopisać ADR-0038 obok
   ADR-0035/36/37).
